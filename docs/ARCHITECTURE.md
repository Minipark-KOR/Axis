# ARCHITECTURE.md

## Project The Axis — Responsibility & Boundary Architecture

***

## 0. 이 문서의 목적

이 문서는 Project The Axis의 **구조적 책임 분리와 경계 규칙을 고정**한다.

이 문서는 다음을 설명하지 않는다.

*   시스템의 철학과 의도 (README의 책임)
*   전체 흐름 개요 (OVERVIEW의 책임)
*   운영 판단 순서 (RUNBOOK / PLAYBOOK의 책임)
*   구현 방법과 코드 구조 (DEVELOPMENT의 책임)

이 문서의 목적은 하나다.

> **어디까지가 누구의 책임이며,  
> 어떤 혼합이 시스템을 즉시 망가뜨리는지를 명시하는 것**

***

## 1. 아키텍처의 최상위 원칙

Axis의 모든 구조 선택은 다음 원칙을 전제로 한다.

1.  제어(Control)와 실행(Execution)은 물리적으로 분리된다
2.  모든 실행은 상태를 보존하지 않는다
3.  실패는 예외가 아니라 정상적인 관측 기록으로 취급된다
4.  시스템은 판단을 수행하지 않는다
5.  구조는 판단 책임을 사람에게 남기기 위해 존재한다

이 원칙은 기능 추가, 확장, 최적화 과정에서도 변경되지 않는다.

***

## 2. 역할별 책임 고정

Axis는 다섯 역할을 가지며, 각 역할의 책임은 서로 침범되지 않는다.

### 2.1 Synapse — Control Plane (Ingress)

**책임**

*   외부 데이터 수집
*   1차 정제 (중복, 형식 오류, 명백한 노이즈 제거)
*   데이터의 분류 및 정리

**금지**

*   장기 저장
*   계산
*   데이터 의미 판단
*   실행 트리거 결정

Synapse는 실행을 “준비”할 뿐,  
실행을 “결정”하거나 “수행”하지 않는다.

***

### 2.2 Neuron — Execution Plane (Compute)

**책임**

*   짧은 시간의 계산 수행
*   실행 결과를 Stage에 기록

**특성**

*   Stateless
*   실행 종료 후 즉시 소멸
*   이전 실행의 상태를 참조하지 않음

**금지**

*   결과 해석
*   결과 전달 시점의 의미 부여
*   다음 실행 조건 판단

Neuron은 계산만 수행하며,  
결과의 의미는 어떤 형태로도 정의하지 않는다.

***

### 2.3 Cortex — Analysis Plane (Memory)

**책임**

*   실행 결과의 누적
*   기록 간 시간적·구조적 관계 보존

**금지**

*   인과 추론
*   중요도 평가
*   자동 관계 생성
*   결론 생성

Cortex는 데이터를 “이해”하지 않는다.  
기록과 맥락을 유지한다.

***

### 2.4 Engram — Storage Plane (Persistence)

**책임**

*   정제된 기록의 장기 보존
*   요청 시 안정적인 재제공

**특성**

*   SSD는 메타데이터 목적에 한정
*   실제 데이터는 HDD 기반 스트리밍

**금지**

*   실행 결정 개입
*   관측 지표 생성
*   상태 요약

Engram은 판단을 돕지 않는다.  
기록의 보관만 책임진다.

***

### 2.5 User — Judgment Plane

**책임**

*   관계의 정의
*   해석의 수행
*   운영 전략 결정

Axis는 사용자의 판단을:

*   검증하지 않고
*   비교하지 않으며
*   자동화하지 않는다

***

## 3. Stage의 아키텍처적 위치

Stage는 저장 계층이 아니라 **구조적 분리 장치**다.

Stage는 Neuron, Cortex, Engram 사이의  
실행 결과를 분리하여 전달하기 위한 **비의미 저장 영역**이다.

구현상 Blob Storage를 사용하지만,  
Stage에 존재하는 데이터는 해석·판단·완성을 의미하지 않는다.

### 3.1 Stage에서 금지되는 것

*   Stage 데이터를 기준으로 한 상태 판단
*   Stage 적재량을 근거로 한 실행 제어
*   Stage 데이터를 “중간 결과”로 해석
*   Stage 데이터에 의미 기반 네이밍 부여

Stage는 **전달 흔적**이지,  
시스템 상태가 아니다.

***

## 4. Control / Execution 경계 규칙

다음 책임은 절대 섞이지 않는다.

*   수집 ↔ 실행
*   실행 ↔ 해석
*   관측 ↔ 판단
*   통계 ↔ 조치
*   기록 ↔ 상태

어떤 기능도 이 경계를 넘는 순간,  
Axis의 구조적 전제는 붕괴된 것으로 간주한다.

***

## 5. 자동화가 들어올 수 없는 지점

다음 위치에는 어떤 형태의 자동화도 허용되지 않는다.

*   실행 중단 결정
*   실행 밀도 조정
*   상태 판정
*   위험도 평가
*   행동 권장 생성

자동화는 실행을 돕는 기술일 수는 있어도,  
판단을 대체하는 구조가 될 수 없다.

***

## 6. 확장 규칙

Axis는 확장을 허용하지만, 다음 조건을 모두 만족해야 한다.

*   기존 역할의 책임을 변경하지 않는다
*   Control / Execution 경계를 침범하지 않는다
*   판단 책임을 시스템으로 이동시키지 않는다

이 조건을 만족하지 못하는 확장은  
기능적 유용성과 무관하게 거부된다.

* Axis의 기본 실행 흐름
(Synapse → Neuron → Cortex → Engram)은 변경되지 않는다.
* 다만, 장기 운용 과정에서 데이터의 재구성 또는 재처리는
새로운 입력 데이터로 취급되어 동일한 기본 흐름을 반복 적용할 수 있다.
* 서버 자원, 병렬성, 실행 밀도는 설계 전제가 아니라 관측 대상이며,
그 효과에 대한 판단은 구조가 아닌 운영 기록과 데이터에 의해 이루어진다

***
