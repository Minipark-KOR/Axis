## Project The Axis — System Overview

## 0. 이 문서의 목적

이 문서는 **Project The Axis의 전체 구조와 작동 방식을 설명**한다.

*   README가 *왜(Why)* 를 고정한다면
*   이 문서는 *무엇이 어떻게 연결되는가(What / How in structure)* 를 설명한다

이 문서는 다음을 다룬다.
- Axis는 전체적으로 어떻게 구성되어 있는가?
- 각 구성 요소는 어떤 책임을 가지는가?
- 데이터와 실행은 어떤 큰 흐름을 따르는가?

이 문서는 다음을 다루지 않는다.
- 운영 판단 기준

운영 흐름과 판단 규칙은 **RUNBOOK / PLAYBOOK**의 책임이며,  
구현 가드는 **DEVELOPMENT.md**의 책임이다.

## 1. Axis의 전체 구조 (Big Picture)

Axis는 
- **제어(Control)와 실행(Execution)을 분리한 분석·운영 시스템**이다.
- 시스템은 데이터를 처리하지만,
그 결과를 **결론·점수·행동으로 변환하지 않는다**.
- 각 단계의 실행 방식(순차/병렬)은 구조 설명에 포함하지 않으며,
운영 규칙에서 별도로 정의한다.

Axis의 목적은 단 하나다.

> 사람이 스스로 판단할 수 있도록  
> **정보의 발자국과 맥락을 시간의 흐름(타임라인)에 따라 기록해두는 것**

***

## 2. Axis의 기본 사고 모델

Axis는 다음 전제를 구조로 고정한다.

*   판단은 계산 결과가 아니라 **사람의 행위**
*   자동화는 실행을 돕되, 판단을 대체하지 않는다
*   시스템은 **상태를 말하지 않고**, 사실만 기록한다

따라서 Axis는 다음을 시스템 내부에 두지 않는다.

*   Health, Risk, Score와 같은 요약 지표
*   정상 / 비정상 상태 값
*   임계 기반 자동 조치
*   자동 추천과 권고 문장

이 전제는 이후 모든 구조 선택의 상위 기준이다.

***

## 3. 전체 구성 요소와 책임 분리

Axis는 서로 다른 책임을 가진 다섯 역할로 구성된다.

### 3.1 Synapse (Control Plane — Ingress)

*   외부 데이터의 **유일한 진입 지점**
*   역할:
    *   데이터 수집
    *   1차 정제 (중복 / 노이즈 제거)
*   하지 않는 것:
    *   장기 저장
    *   계산
    *   판단

Synapse의 역할은 분석이 아니라
정보와 실행을 분리하기 위한 제어면이다.

***

### 3.2 Neuron (Execution Plane — Compute)

*   짧은 시간의 계산을 수행하는 **단기 실행 서버**
*   특징:
    *   상태 비보존(Stateless)
    *   작업은 blob stage에 기록된 데이터(각 작업별로 수행)
    *   실행 종료 후 즉시 소멸
*   결과 처리:
    *   결과를 *의미 없이* stage에 기록
    *   완료 시 작업별 fetch 신호만 전달

Neuron은 **결과를 해석하지 않고**,  
다음 판단 시점을 시스템에 강제하지 않는다.

***

### 3.3 Cortex (Analysis Plane — Memory)

*   Axis에서 **전체 기록을 바라볼 수 있는 누적 계층**
*   역할:
    *   실행결과와 기록을 누적
    *   기록 간의 시간적·구조적 관계 보존
*   하지 않는 것:
    *   인과 추론
    *   중요도 평가
    *   결론 생성

Cortex는 판단하지 않는다.  
**데이터를 기록하고,기록과 기록간의 맥락을 유지한다.**

***

### 3.4 Engram (Storage Plane — Persistence)

*   정제된 기록의 **장기 생존 책임자**
*   역할:
    *   장기 보관
    *   요청 시 빠른 응답
*   특징:
    *   SSD는 메타데이터 전용
    *   실제 데이터는 HDD 기반 스트리밍

Engram은 “빠른 시스템”이 아니라  
**기록이 오래 유지되는 시스템**이다.

***

### 3.5 User (Judgment Plane)

*   Axis에서 **유일한 판단 주체**
*   사용자가 결정한다:
    *   무엇을 연결할지
    *   어떤 관계를 추적할지
    *   언제 운영 전략을 바꿀지

Axis는 사용자의 판단을:

*   평가하지 않고
*   추천하지 않으며
*   자동화하지 않는다

***

## 4. 제어와 실행의 분리 구조

Axis는 다음 두 계층을 물리적으로 분리한다.

### 4.1 Control Plane

*   상시 가동
*   데이터 수집 및 준비
*   실행 가능 조건 판단

### 4.2 Execution Plane

*   필요할 때만 생성
*   실행 후 즉시 종료
*   실패가 누적되지 않음

이 분리는 다음을 보장한다.

*   실행 실패가 시스템 안정성에 영향을 주지 않음
*   비용과 실행 밀도를 독립적으로 제어
*   고사양 자원의 상시 점유 방지

***

## 5. 데이터 흐름 개요

기본 흐름은 아래와 같다.

    External Source
       ↓
    Synapse (수집 / 1차 정제)
       ↓
    Stage / Blob(Neuron)
       ↓
    Neuron (단기 계산)
       ↓
    Stage / Blob(Cortex)
       ↓
    Cortex (기억 / 관계 누적)
       ↓
    Stage / Blob(Engram) | Stage / HDD(Engram)
       ↓
    Engram (장기 보관)

이 흐름은 다음 상황에서도 **변하지 않는다**.

*   실행 실패
*   부분 완료
*   재시도

* 기본 실행 순서(Synapse → Neuron → Cortex → Engram)는 항상 유지된다.
* 데이터가 축적된 이후에도 이 순서는 변경되지 않으며,
재처리는 기존 결과를 수정하지 않고 새로운 입력을 생성하는 방식으로만 수행된다.

* Stage는 Neuron, Cortex, Engram 사이의
실행 결과를 분리하여 전달하기 위한 비의미 저장 영역이다.
* 구현상 Blob Storage를 사용하지만,
Stage에 존재하는 데이터는 해석·판단·완성을 의미하지 않는다.

***

## 6. 실패를 다루는 구조적 이유

Axis에서 실패는 에러가 아니라 **관측 데이터**다.

*   단일 실패는 즉시 조치 대상이 아니다.
*   반복된 실패 패턴만이 운영 판단의 대상이 된다.
*   실패는 기록으로 남고, 운영자가 판단한다.
*   자동 복구는 존재하지 않음

실패를 제거하지 않기 때문에,
Axis는 실패 이력 자체를 **운영 사고의 입력**으로 유지할 수 있다.

***

## 7. 자동화를 제한한 구조적 선택

Axis는 다음 자동화를 구조적으로 배제한다.

*   자동 스케일
*   자동 중단
*   조건 기반 실행 조정
*   상태 요약 지표

이 선택은 기술 한계 때문이 아니다.

> 자동화가 들어오는 순간  
> 판단 책임이 시스템 쪽으로 이동하기 때문이다.

Axis는 이 이동을 끝까지 거부한다.

Axis는
