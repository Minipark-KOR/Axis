# Archived Decisions

## 0. Purpose

본 문서는 Axis 시스템에서 **이미 결정되었으며,
재논의의 대상이 되지 않는 설계·운영 판단**을 기록한다.

이 문서의 목적은 다음과 같다.

- 과거 결정의 맥락을 보존한다
- 동일한 판단을 반복하지 않도록 한다
- 현재 규범 및 기술 문서의 배경을 설명한다

이 문서는 다음을 하지 않는다.

- 새로운 규칙을 제안하지 않는다
- 기존 문서의 내용을 재정의하지 않는다
- 현재 운영 판단의 근거로 사용되지 않는다

이 문서는 **참고 기록**이며,
현재 규범의 근거로 인용되지 않는다.

## 1. Execution Model Decisions

### 1.1 Single Pipeline with Re-entry

Axis는 다음 기본 실행 순서를 채택한다.

Synapse → Neuron → Cortex → Engram

이 순서는 변경하지 않으며,
모든 데이터는 이 흐름을 따른다.

장기 운용 과정에서 데이터가 재처리되는 경우에도,
이는 기존 결과의 수정이 아니라
**새로운 입력을 생성하여 동일한 파이프라인을 반복 적용하는 방식**으로 수행한다.

이 결정은 다음 목적을 가진다.

- 실행 구조의 단순화
- 데이터 출처의 명확화
- 과거 결과의 불변성 유지

## 2. Resource and Infrastructure Decisions

### 2.1 Initial Server Class Unification

초기 운영 단계에서,
Neuron과 Cortex 서버는 동일한 서버 클래스를 사용하기로 결정하였다.

이 결정의 목적은 다음과 같다.

- 병렬성 및 코어 수 증가의 효과를
  관측 대상으로 명확히 분리하기 위함
- 성능 부족 여부에 대한 의심을
  초기 단계에서 제거하기 위함
- 이후 하향 조정 결정을
  데이터에 기반해 단순화하기 위함

서버 클래스 통일은
최적화가 아니라 **관측을 위한 기준선**으로 채택되었다.

### 2.2 Separation of Compute and Storage Investment

Engram 서버는 다음 선택을 확정하였다.

- 계산 자원은 최소화한다
- 저장 자원은 비교적 넉넉하게 유지한다

이는 다음 전제를 따른다.

- 계산은 비용 변동성이 크다
- 저장은 장기 비교와 재해석의 자유도를 제공한다

따라서 남는 자원은
추가 계산이 아니라 **기록 보존**에 우선적으로 사용한다.

## 3. Operation Rhythm Decisions

### 3.1 Adoption of Fixed Execution Rhythm

Axis는 실행 리듬을
결과에 따라 자동으로 변화시키지 않기로 결정하였다.

- 실행 실패는 다음 실행 시점을 변경하지 않는다
- 실행 결과는 리듬 조정의 직접적 트리거가 아니다
- 리듬 변경은 명시적 운영 판단으로만 수행된다

이 결정은 다음을 보장하기 위한 것이다.

- 장기 비교의 안정성
- 운영 예측 가능성
- 자동 판단 유입 차단

### 3.2 Non-Continuous Execution

Axis는 상시 실행 모델을 채택하지 않는다.(Synapse 제외, 수집은 상시)

- 개별 실행은 짧게 종료된다
- 실행 간 상태는 보존하지 않는다
- 항상 종료 가능한 구조를 유지한다

이는 안정성을 높이기 위한 선택이 아니라,
**판단 책임을 시스템에 축적하지 않기 위한 설계 선택**이다.

## 4. Data Retention and Reference Decisions

### 4.1 Purge-Marker Based Deletion

Axis는 즉시 삭제를 허용하지 않는다.

- 삭제는 항상 purge-marker 단계를 거친다
- purge-marker는 상태가 아니라 태그다
- 유예 기간 중 참조가 발생하면 즉시 복귀한다

이 결정은
의도치 않은 데이터 손실을
구조적으로 차단하기 위한 선택이다.

### 4.2 Conservative Reference Interpretation

Axis는 참조(reference)를
삭제를 방지하는 방향으로만 해석한다.

- 애매한 경우에는 참조로 간주한다
- 단 한 번의 참조도 생존을 연장한다
- 참조 횟수는 판단 기준이 아니다

이는 데이터 보존의 안전성을
기본값으로 두기 위한 결정이다.

## 5. Non-Adoption Decisions

다음 항목들은 검토되었으나,
의도적으로 채택하지 않기로 결정되었다.

- 자동 비용 제어
- 자동 실행 축소 또는 확장
- 토큰 수치 기반 자동 판단
- 상태 점수 또는 건강도 지표
- 시스템 내부 추천 또는 권고 메커니즘

이 결정들은 기술적 한계가 아니라,
**판단 책임을 사람에게 남기기 위한 선택**이다.

## 6. Final Note

본 문서에 기록된 결정들은
현재 Axis 문서 체계의 배경이 되는 판단들이다.

이 문서는 규칙을 정의하지 않으며,
현재 규범을 수정하지 않는다.

이 문서의 역할은
“왜 이렇게 되었는가”를 기억하는 데 그친다.
과거의 결정은 과거의 맥락에서 이루어진 판단이므로,
현재의 판단과는 독립적으로 이해되어야 한다.
