# Operation Rhythm

## 0. Scope

본 문서는 Axis 시스템의 **운영 리듬(operation rhythm)**을 정의한다.
여기서 리듬이란,
각 실행 단계가 **언제, 어떤 주기로 활성화되는지**를 의미한다.

이 문서는 다음을 정의한다.
- 실행이 발생하는 시간적 패턴
- 실행이 격리되는 단위와 호흡
- 단계별 실행 리듬의 관계

이 문서는 다음을 정의하지 않는다.
- 실행 여부에 대한 판단 기준
- 실행 결과의 해석
- 비용 또는 성능에 대한 권고
- 자동 스케줄 조정 로직

리듬은 결정을 대신하지 않는다.
리듬은 **관측 가능한 반복 구조**를 만든다.

## 1. Principle of Rhythm

Axis의 모든 실행은
**짧고 명확한 단위**로 나뉘어 수행된다.

Axis는 다음 원칙을 따른다.

- 실행은 장시간 지속되지 않는다
- 실행 간 상태를 축적하지 않는다
- 실행 리듬은 예측 가능해야 한다

리듬의 목적은 성능 극대화가 아니라
**관측과 통제의 단순화**다.

## 2. Base Execution Cycle

Axis 운영의 기본 시간 단위는
**6시간 주기**다.

하루는 다음 네 개의 실행 구간으로 나뉜다.

- 00:00
- 06:00
- 12:00
- 18:00

각 주기는 서로 독립적이며,
하나의 주기에서 발생한 실행 결과는
다음 주기의 실행 조건을 변경하지 않는다.

## 3. Execution Slots

각 실행 주기 내에서,
실행은 하나 이상의 **슬롯(slot)**으로 나뉘어 수행될 수 있다.

슬롯은 다음 특성을 가진다.

- 슬롯은 짧은 실행 단위다
- 슬롯 간 상태는 공유되지 않는다
- 슬롯은 실패해도 다음 슬롯에 영향을 주지 않는다

슬롯의 개수와 간격은
운영 설정의 책임이며,
본 문서에서는 고정하지 않는다.

## 4. Stage-Level Rhythm

Axis의 각 서버 유형은 서로 다른 운영 리듬을 가진다.
이 리듬은 성능 최적화를 위한 장치가 아니라,
실행 시점과 관측 기준을 명확히 하기 위한 운영 구조다.

### 4.1 Synapse

Synapse 서버는 상시 운영된다.

- 외부 입력 데이터의 수집
- 1차 정제 및 분리
- 의미 단위로의 구조화

Synapse는 실행 주기를 가지지 않으며,
외부 입력의 흐름에 따라 지속적으로 동작한다.


### 4.2 Neuron / Cortex

Neuron 및 Cortex 서버는
정해진 시간 창 내에서만 주기적으로 실행된다.

#### 비가동 구간

- 매일 02:00부터 06:00까지 서버는 종료 상태를 유지한다
- 해당 시간 동안 실행은 발생하지 않는다

#### 가동 구간

- 06:00부터 다음 날 02:00까지 실행 가능
- 하루 총 20회 실행

#### 실행 시점

- Neuron: 매 정시 5분
- Cortex: 매 정시 10분  
  (항상 Neuron 실행 이후 5분 지연)

#### 실행 특성

- 각 실행은 독립적이다
- 실행 후 즉시 종료된다
- 최대 허용 실행 시간은 다음과 같다:
  - 계산(runtime): 120초
  - 후처리: 15초  
    (저장, 파일 이동, 점검, 무결성 검사 등)
- Neuron은 계산 전용으로 설계되었다.
- Cortex는 후처리를 중심으로 하며 데이터간의 관계를 설정하고, 전체적인 데이터 흐름을 조율한다.

동일 주기 내에서
복수의 독립 실행이 허용되며,
이전 실행의 상태는 다음 실행으로 전달되지 않는다.

### 4.3 Engram

Engram 서버는 상시 운영되지 않는다.

Engram의 운영 리듬은
**6시간 주기**로 고정된다.

하루는 다음 네 개의 Engram 실행 구간으로 나뉜다.

- 02:20
- 08:20
- 14:20
- 20:20

각 실행은 다음 특성을 가진다.

- 짧은 시간 동안만 활성화된다
- 누적된 데이터의 집계와 정리를 수행한다
- 실행 종료 후 서버는 다시 종료 상태로 전환된다

Engram은 다른 단계의 실행을 트리거하지 않으며,
다른 단계의 실행 결과에 의해 스스로 실행 시점을 변경하지 않는다.

## 5. Rhythm Isolation Rule

Axis에서 운영 리듬은 다음 규칙에 의해 유지된다.

- 한 단계의 실행 실패는 다른 단계의 리듬에 영향을 주지 않는다
- 실행 성공 또는 실패는 다음 실행 시점을 변경하지 않는다
- 리듬 변경은 자동으로 발생하지 않는다

리듬은 운영자가 명시적으로 설정한 구조이며,
실행 결과로부터 스스로 진화하지 않는다.

## 6. Relationship to Observation

리듬은 관측을 위한 기준선 역할을 한다.

- 동일한 리듬 위에서 실행 결과를 비교할 수 있다
- 리듬은 장기 추세를 관측하기 위한 좌표를 제공한다
- 리듬 자체는 성능 평가 대상이 아니다

리듬은 결과를 해석하기 위한 조건이지,
결과로부터 수정되는 대상이 아니다.

## 7. Stability and Change

운영 리듬은 다음 특성을 갖는다.

- 장기간 유지됨을 전제로 설계된다
- 변경은 예외적인 운영 판단에 의해 이루어진다
- 변경 시에도 기존 데이터의 의미는 변하지 않는다

리듬은 자주 조정되는 설정이 아니라,
**비교를 가능하게 하는 고정점**이다.
