# Cost Control Guardrails

## 0. Scope

본 문서는 Axis 시스템의 **비용 통제 가드레일(cost control guardrails)**을 정의한다.
가드레일이란, 운영을 자동으로 조정하기 위한 규칙이 아니라
**비용이 예측 가능하도록 제한선을 명확히 하는 운영 기준**을 의미한다.

이 문서는 다음을 정의한다.
- 서버 단위의 월간 비용 상한
- 실행 시간 예산의 계산 근거
- 비용 초과를 방지하기 위한 구조적 제한

이 문서는 다음을 정의하지 않는다.
- 자동 실행 중단 또는 자동 축소 로직
- 비용 효율에 대한 평가
- 운영 전략 변경에 대한 권고
- 성능 최적화 판단

비용 통제는 자동화의 대상이 아니라,
**명시적으로 설계된 제약 조건**이다.

## 1. Cost Control Principle

Axis의 비용 통제는 다음 원칙을 따른다.

- 비용은 사후 분석 대상이 아니라 설계의 출발점이다
- 시스템은 비용을 넘어서는 동작을 시도하지 않는다
- 비용 초과에 대한 판단은 시스템이 아닌 사람이 수행한다

가드레일은 비용을 절약하기 위한 장치가 아니라,
비용이 불확실해지는 상황을 구조적으로 차단하기 위한 장치다.

## 2. Monthly Budget Cap

각 Azure 서버 인스턴스는
월간 비용 상한을 가진다.

- 서버당 월 예산 상한: **USD 8**
- 예산은 서버 단위로 독립적으로 적용된다
- 예산의 이월 또는 공유는 허용되지 않는다

이 상한은 경고 임계값이 아니라,
설계 시점에 고정된 운영 제약이다.

## 3. Execution Time Budget

월 예산 상한과 시간 단가는
서버의 월간 실행 가능 시간을 결정한다.

- 기준 서버: Azure F4s_v2
- 기준 단가: USD 0.376 / hour
- 월 예산 상한: USD 8

이를 기준으로 계산되는
이론적 최대 실행 시간은 약 **21시간 / 월**이다.

## 4. Daily Time Cap

월간 실행 시간 예산을
일 단위로 분산하면 다음과 같다.

- 일 평균 허용 시간: 약 42.5분
- 운영 안전 여유를 고려한
  **일일 최대 허용 실행 시간: 40분**

이 기준에서는
월 최대 실행 시간이 약 20시간에 해당하며,
예산 상한 대비 여유를 유지한다.

이 일일 시간 제한은
성능 목표가 아니라 비용 안정성을 위한 기준선이다.

## 5. Relationship to Execution Rhythm

비용 통제 가드레일은
운영 리듬과 결합되어 적용된다.

- 실행 슬롯의 개수와 간격은
  일일 시간 예산을 초과하지 않도록 설계된다
- 개별 실행은 짧고 독립적으로 유지된다
- 단일 실행의 성공 또는 실패는
  비용 가드레일을 변경하지 않는다

비용 가드레일은
실행을 평가하지 않고,
실행의 총량만을 제한한다.

## 6. Non-Automation Rule

다음 동작은 명시적으로 금지된다.

- 비용 또는 시간 소진을 기준으로 한 자동 실행 중단
- 비용 소모 패턴에 따른 자동 실행 축소
- 비용 수치를 상태 또는 점수로 사용하는 로직

비용 초과를 감지하더라도,
그에 대한 대응은 항상
운영자의 명시적 판단과 설정 변경을 통해 이루어진다.

## 7. Stability and Revision

비용 가드레일은
장기간 유지됨을 전제로 설계된다.

- 가드레일은 빈번히 조정되지 않는다
- 변경이 필요할 경우,
  변경 내용과 시점은 명시적으로 기록된다
- 과거 데이터의 의미는 변경되지 않는다

가드레일은 튜닝 파라미터가 아니라,
운영 조건의 일부다.
