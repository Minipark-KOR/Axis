## DEVELOPMENT.md

이 문서는 **가장 마지막에 와야 하는 문서**다.  
이유는 명확하다.

> 모든 사고·구조·운영 규칙이  
> **코드에서 무너질 수 있기 때문**이다.

이 문서는

*   구현 자유도를 설명하지 않고
*   코드 스타일 가이드를 늘어놓지 않으며
*   기능을 추가하는 법을 말하지 않는다

**오직 “이 선을 넘으면 Axis가 아니다”만 고정**한다.

아래는 **그대로 `DEVELOPMENT.md`로 확정 가능한 최종본**이다.

# DEVELOPMENT.md

## Project The Axis — Implementation Guardrails

***

## 0. 이 문서의 목적

이 문서는 Project The Axis의 **구현 단계에서 절대 침범해서는 안 되는 가드레일**을 정의한다.

이 문서는 다음을 설명하지 않는다.

*   기능 명세
*   API 사용 방법
*   성능 최적화 전략
*   언어나 프레임워크 선택

이 문서의 목적은 하나다.

> **구현이 철학·구조·운영 판단을 침식하지 않도록  
> 코드 차원에서 경계를 고정하는 것**

이 문서는 다음 상위 문서를 전제로 한다.

*   README.md
*   OVERVIEW\.md
*   ARCHITECTURE.md
*   RUNBOOK.md
*   STATISTICS.md

개발은 이 문서들을 **해석하지 않는다**.  
개발은 이 문서들을 **전제로만 한다**.

***

## 1. 구현의 최상위 원칙

모든 코드는 다음 원칙을 만족해야 한다.

1.  구현은 판단을 수행하지 않는다
2.  구현은 해석을 포함하지 않는다
3.  구현은 실행을 자동화하지 않는다
4.  구현은 구조적 책임을 넘지 않는다
5.  구현은 언제든 제거 가능해야 한다

코드가 “똑똑해질수록”,  
Axis는 망가진다.

***

## 2. 의존성 방향 규칙 (강제)

Axis의 모든 의존성은 **단방향**이다.

    api  →  flows  →  domain  →  infrastructure

### 규칙

*   상위 레이어는 하위 레이어를 참조할 수 있다
*   하위 레이어는 상위 레이어를 **절대** 참조할 수 없다

### 금지 예시

*   domain이 api 상태를 확인함
*   infrastructure 코드에서 판단 플래그를 생성함
*   flows에서 통계 결과를 해석하여 분기함

의존성 역전은  
**판단 책임의 역전**으로 간주한다.

***

## 3. Public API 규칙

### 정의

*   `api/` 하위만 외부 진입점이다
*   `flows/`, `domain/`, `infrastructure/`는 직접 호출 불가

### 금지

*   Bot 또는 외부 시스템이 domain 함수를 직접 호출
*   테스트 코드가 infrastructure 구현을 우회 호출

API는 구현을 숨기기 위한 장치가 아니다.  
**판단 이동을 막기 위한 경계**다.

***

## 4. 구현 레이어별 금지 사항

### 4.1 flows/

**역할**

*   순서 정의
*   연결
*   orchestration

**금지**

*   계산 로직
*   상태 판단
*   조건 기반 실행 변경

flows는 “왜”를 묻지 않는다.  
“어떤 순서로”만 다룬다.

***

### 4.2 domain/

**역할**

*   계산
*   의미 구조 정의
*   테스트 중심 로직

**금지**

*   외부 시스템 인지
*   실행 제어
*   판단 문구 생성
*   추천 로직

domain은 맞고 틀림을 계산할 수는 있어도,  
좋고 나쁨을 말해서는 안 된다.

***

### 4.3 infrastructure/

**역할**

*   IO
*   외부 통신
*   기술적 연결

**금지**

*   의미 부여
*   조건 판단
*   로직 분기

infrastructure는  
가장 멍청해야 한다.

***

## 5. 관측 및 로그 구현 규칙

### 원칙

*   로그는 사실만 기록한다
*   평가·의견·권고 문장은 금지
*   성공·실패 총평 금지

### 금지 예시

*   “처리가 느립니다”
*   “정상적으로 완료되었습니다”
*   “재시도가 필요합니다”

로그는  
사람이 읽기 위한 문장이 아니라,  
판단을 위한 **원본 기록**이다.

***

## 6. 통계 결과 사용 금지 규칙

다음 행위는 코드 리뷰 없이 즉시 반려된다.

*   통계 결과로 실행 분기
*   평균·점수를 조건문에 사용
*   통계를 상태 플래그로 변환
*   기준선 변경을 자동화

통계는  
**if 조건이 아니다.**


다음 구현 또한 명시적으로 금지된다.

* 토큰 수치, 데이터 크기, 실행 시간 등의 관측 값을 기준으로
  실행 밀도, 병렬성, 서버 사양을 자동으로 변경하는 로직
* 관측 데이터를 내부 점수, 등급, 상태 값으로 환원하는 구현
* 자원 결정은 항상 사람의 판단과 외부 설정 변경을 통해 이루어져야 한다.

***

## 7. 테스트 철학

### 테스트의 목적

*   결과의 정답 여부 확인하지 않는다.
*   책임 위치를 확인한다.
*   판단·해석·자동화가 없는지 감시한다.

### 테스트가 반드시 답해야 하는 질문

*   이 로직은 판단을 수행하고 있는가?
*   이 조건은 경계를 침범하고 있는가?
*   이 변경은 되돌릴 수 있는가?

테스트는 신뢰를 보장하는 게 아니라,  
**월권을 감시**해야 한다.

***

## 8. 금지 패턴 요약

다음 구현은 Axis에서 즉시 거부된다.

*   자동 실행 조정
*   자동 상태 판정
*   자동 권고 생성
*   점수 기반 분기
*   침묵을 판단으로 해석하는 로직

편리함은  
Axis의 목표가 아니다.

***

## 9. 요약

DEVELOPMENT.md는  
“어떻게 구현할 것인가”를 말하지 않는다.

이 문서는  
**어떤 구현이 Axis를 깨뜨리는지**만 고정한다.

코드는 언제든 바꿀 수 있다.  
구조와 판단 책임은 바꿀 수 없다.
