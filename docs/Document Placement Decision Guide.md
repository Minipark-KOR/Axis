# Axis 문서 위치 판단 가이드

Document Placement Decision Guide

이 문서는 **새 문서를 작성할 때 그 문서를 어디에 두어야 하는지 결정하기 위한 판단 가이드**다.  
문서의 품질·내용·완성도를 평가하지 않으며, **문서의 역할에 따라 위치만 결정한다.**

이 가이드는 규칙 문서가 아니라 **판단 트리**다.  
아래 질문을 **위에서부터 순서대로** 적용한다.  
**처음으로 YES가 되는 지점이 해당 문서의 위치다.**

***

## 판단 절차

### Q1. 이 문서가 Axis의 정체성, 철학, 판단 책임의 위치를 정의하거나 변경하는가?

다음 중 하나라도 해당하는가:

*   시스템이 무엇을 하는지 또는 하지 않는지를 새로 규정한다
*   판단 주체가 어디에 있는지를 바꾼다
*   Control / Execution / Judgment의 경계를 재정의한다
*   Axis의 철학이나 헌장을 보강·수정한다

**YES → 이 문서는 새로 만들지 않는다**

해당 역할의 문서는 이미 완료되었으며,  
새 문서를 추가하는 순간 Axis의 정체성이 흔들린다.

**NO → Q2로 이동**

***

### Q2. 이 문서가 형식, 규칙, 표준을 고정하는 문서인가?

다음 성격을 가지는가:

*   네이밍 규칙
*   포맷 또는 스키마
*   고정된 문자열, 코드, 구조
*   변경 시 파급 범위가 명확한 규칙
*   판단·해석을 포함하지 않는다

**YES → `docs/specs/`**

예시:

*   Metric 네이밍 규칙
*   Event / Error Code 표준
*   Operational Identifier 규칙
*   로그 포맷 규칙

**NO → Q3로 이동**

***

### Q3. 이 문서가 운영 중 사람이 무엇을 보고 어떤 순서로 판단해야 하는지를 다루는가?

다음 성격을 가지는가:

*   운영 중 확인 질문 목록
*   판단의 순서와 흐름
*   실행이나 조치를 지시하지 않는다
*   결론이나 권고를 포함하지 않는다

**YES → `docs/ops/`**

예시:

*   운영 체크리스트
*   Playbook
*   상황별 점검 시나리오

**NO → Q4로 이동**

***

### Q4. 이 문서가 Control Plane, Bot, UI의 기술적 역할과 제약을 설명하는가?

다음 성격을 가지는가:

*   Bot 또는 제어 인터페이스의 책임 설명
*   Slack / Email / UI의 전달 규칙
*   제어면 구조와 기술적 제약
*   운영 판단이나 시스템 철학은 다루지 않는다

**YES → `docs/control-plane/`**

예시:

*   BOT\_ARCHITECTURE.md

**NO → Q5로 이동**

***

### Q5. 이 문서가 과거의 설계 이유나 맥락을 기록하는 문서인가?

다음 성격을 가지는가:

*   왜 이런 선택을 했는지 설명한다
*   초기 통합 문서 또는 설계 초안이다
*   과거 운영 가이드나 실험 기록이다
*   현재 규칙의 배경을 설명한다

**YES → `docs/history/`**

주의:

*   읽는 것은 허용된다
*   현재 규칙의 근거로 인용하는 것은 금지된다

**NO → Q6로 이동**

***

### Q6. 어느 위치에도 명확히 들어가지 않는가?

**YES → 문서 성격이 잘못 설계되었다**

선택지는 다음 중 하나다:

*   문서를 분해한다
*   질문을 다시 정의한다
*   문서를 만들지 않는다

Axis에서는 **역할이 불분명한 문서**를 허용하지 않는다.

***

## 문서 생성 중단 신호

다음 생각이 들면 문서 작성을 즉시 중단해야 한다:

*   “일단 notes로 둔다”
*   “나중에 정리할 가이드”
*   “best practices를 모아두면 좋겠다”
*   “README에 조금만 추가하자”

이 신호는 **판단 책임이 문서로 이동하고 있다는 명확한 경고**다.

***

## 요약

문서의 위치는 내용의 수준이나 중요도가 아니라 **역할**로 결정된다.  
역할이 분명하지 않다면, 문서 자체가 잘못된 것이다.
